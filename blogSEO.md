# 블로그 작성 지침

앞으로 개발 내용을 블로그용으로 정리할 때는, 단순한 작업 로그나 기술 문서처럼 쓰지 말고 `내 블로그 말투`와 `SEO를 고려한 구조`를 함께 반영해서 작성한다.

이 문서는 내가 이후에 "블로그용으로 정리해줘", "포스팅 스타일로 써줘", "SEO 스타일로 정리해줘" 같은 요청을 했을 때 따를 기준이다.

## 1. 글의 기본 목적

이 글의 목적은 단순히 코드 결과를 나열하는 것이 아니라,  
`왜 이렇게 구현했는지`, `구조를 어떻게 나눴는지`, `현재 어디까지 왔는지`를 독자가 이해할 수 있도록 설명하는 것이다.

즉, 아래 3가지를 동시에 만족해야 한다.

- 내 블로그 스타일처럼 설명형 문체를 유지할 것
- 검색 유입을 고려해 핵심 키워드를 자연스럽게 녹일 것
- 코드 전체를 다 보여주기보다 핵심 구현 포인트를 설명 중심으로 정리할 것

## 2. 말투와 문체

전체 문체는 너무 딱딱한 기술 문서체가 아니라,  
누군가에게 개념을 알려주듯이 풀어서 설명하는 `설명형 블로그 스타일`로 작성한다.

### 지켜야 하는 톤

- 독자에게 설명해주는 느낌으로 쓸 것
- 단순 정의보다 "왜 이 구조가 필요한지"를 같이 설명할 것
- 문장을 너무 짧게 끊지 말고 자연스럽게 이어서 설명할 것
- 너무 학술적이거나 지나치게 딱딱한 어조는 피할 것
- 너무 가볍거나 장난스러운 말투도 피할 것

### 자연스럽게 사용할 수 있는 연결 표현

- 쉽게 말하면
- 결국
- 즉
- 왜냐면
- 따라서
- 이 구조를 쓰는 이유는
- 여기서 중요한 것은

### 피해야 하는 톤

- 문장마다 bullet만 나열하는 문서식 표현
- 지나치게 명령조인 문체
- 사전식 정의만 반복하는 설명
- "이 함수는 A를 한다. 다음 함수는 B를 한다." 식의 단순 나열

## 3. 글의 전체 구조

글은 가능하면 아래 흐름을 따른다.

1. 이번 작업의 목적
2. 왜 이 구조 또는 기능이 필요한지
3. 실제 파일 또는 계층 역할 설명
4. 구현 흐름 설명
5. 핵심 코드 발췌
6. 현재 결과
7. 다음 단계

즉 `개념 -> 구조 -> 구현 흐름 -> 핵심 코드 -> 결과` 순서로 읽히게 만든다.

## 4. 글 도입부 작성 방식

도입부에서는 바로 코드 설명부터 들어가지 말고,  
이번 작업이 어떤 맥락에서 필요한 작업인지 먼저 설명해야 한다.

예를 들면:

- 이번에는 Flame 점프게임에서 플레이어가 실제로 움직일 수 있도록 이동, 점프, 중력 구조를 붙인 과정
- 이번 작업에서는 단순히 초록 네모를 띄우는 것이 아니라, 이후 적, 함정, 발판 시스템까지 붙일 수 있는 방식으로 플레이어 구조를 먼저 잡은 것

핵심은 "이번 글에서 무엇을 다룰지"를 초반에 분명하게 보여주는 것이다.

## 5. SEO 기준

SEO를 고려하되, 검색어를 억지로 반복하는 방식은 피한다.  
핵심은 검색 키워드를 문맥 안에 자연스럽게 녹여서, 글이 사람에게도 읽히고 검색에도 걸리도록 만드는 것이다.

### 핵심 키워드 예시

상황에 따라 아래 키워드를 자연스럽게 사용한다.

- Flutter Flame 점프게임
- Flame 플랫폼 게임
- Flame 플레이어 이동 구현
- Flame 점프 구현
- Flame 키보드 입력 처리
- Flame GameWidget
- Flutter Flame Windows 실행

### 키워드 사용 원칙

- 제목에 핵심 키워드 1개 이상 포함
- 첫 문단 안에 핵심 키워드 1개 이상 포함
- 중간 소제목에도 검색어를 의식한 표현 포함
- 본문 전체에서 2~4회 정도 자연스럽게 반복
- 같은 키워드를 부자연스럽게 연속 반복하지 말 것

### 좋은 예

- `Flutter Flame 점프게임에서 Player 이동과 점프를 먼저 구현한 이유`
- `Flame 플랫폼 게임 구조에서 InputSystem을 따로 분리한 이유`

### 피해야 하는 예

- `Flame 점프게임 Flame 플랫폼 게임 Flame Player 이동 Flame 입력 처리`

## 6. 제목 작성 방식

제목은 가능한 한 `무엇을 구현했는지 + 어떤 기술인지`가 같이 드러나게 쓴다.

예시:

- `Flutter Flame 점프게임에서 Player 이동과 점프를 먼저 구현한 이유`
- `Flame 플랫폼 게임 구조에서 JumpGame, Level, Player를 나눈 이유`
- `Flame 키보드 입력 처리와 플레이어 이동 구조 정리`

## 7. 소제목 작성 방식

소제목은 너무 추상적으로 쓰지 말고,  
검색 키워드와 설명 목적이 동시에 드러나게 쓴다.

좋은 예:

- `Flame에서 JumpGame을 먼저 만든 이유`
- `Level에서 바닥을 먼저 만든 이유`
- `Player에서 이동과 점프를 직접 처리한 이유`
- `InputSystem을 따로 둔 이유`

피해야 하는 예:

- `설명`
- `기능`
- `코드`

## 8. 코드 사용 원칙

코드는 전체를 전부 넣지 말고, `핵심 구현 포인트가 드러나는 부분만 발췌`한다.

### 코드 발췌 우선순위

- 진입점 연결 코드
- 구조를 나누는 핵심 코드
- 이동/점프/입력/충돌처럼 실제 동작의 중심 코드
- 설계 의도가 드러나는 코드

### 굳이 안 넣어도 되는 것

- import 전체
- 비어 있는 클래스
- 단순 보일러플레이트
- 의미 없는 getter/setter
- 아직 사용하지 않는 placeholder 코드

### 코드 아래 설명 방식

코드를 보여준 다음에는 반드시 아래를 다시 설명한다.

- 이 코드가 무슨 역할을 하는지
- 왜 이런 구조로 작성했는지
- 이후 어떤 확장을 위해 이렇게 나눴는지

즉, `코드 -> 설명` 순서를 반드시 지킨다.

## 9. 파일/구조 설명 방식

파일을 설명할 때는 단순히 "이 파일은 무엇을 한다"에서 끝내지 않는다.  
아래 3가지를 같이 설명한다.

1. 이 파일이 맡는 역할
2. 왜 따로 분리했는지
3. 나중에 무엇을 확장하기 쉬워지는지

예시:

- `InputSystem`은 단순히 키보드 입력을 받기 위한 파일이 아니라, 나중에 터치 입력이나 게임패드 입력이 들어오더라도 Player 로직을 바꾸지 않기 위해 따로 분리한 구조라고 설명할 것
- `Level`은 아직 단순한 바닥 하나만 그리지만, 이후 발판, 스프링, 체크포인트 같은 월드 오브젝트를 붙이기 위한 월드 루트 역할이라고 설명할 것
- `Player`는 단순 초록 네모가 아니라, 이동/점프/중력/착지 상태를 관리하는 핵심 엔티티라고 설명할 것

## 10. 설명의 초점

글의 중심은 `무엇을 만들었는가`보다 `왜 이렇게 만들었는가`에 둔다.

따라서 아래 질문에 답하는 식으로 쓴다.

- 왜 `main.dart`를 `GameWidget<JumpGame>`으로 바꿨는가
- 왜 `JumpGame`에서 `Level`과 `Player`를 연결했는가
- 왜 입력과 플레이어 행동을 분리했는가
- 왜 처음에는 카메라 추적을 제거했는가
- 왜 이동 수치를 상수로 분리했는가

## 11. 결과 정리 방식

글 마지막에는 반드시 현재 구현 결과를 요약한다.

예시:

- 현재는 초록 사각형 플레이어가 바닥 위에서 좌우 이동과 점프가 가능한 상태까지 구현했다
- 키보드 입력은 A/D, 방향키, Space 기준으로 동작하도록 연결했다
- 아직 실제 스프라이트와 타일 충돌은 붙이지 않았고, 우선 이동 감각을 확인할 수 있는 단계까지 왔다

## 12. 다음 단계 연결 방식

마무리는 항상 다음 작업과 이어지게 한다.  
그래야 글이 개발 흐름 안에서 자연스럽게 이어진다.

예시:

- 다음 단계에서는 초록 네모 대신 실제 캐릭터 스프라이트를 붙일 예정
- 다음에는 바닥 하나가 아니라 여러 플랫폼을 배치하고 충돌 구조를 확장할 예정
- 이후에는 코인과 체크포인트를 붙여서 실제 점프게임 흐름에 가깝게 만들 예정

## 13. 출력 형식

- 기본은 Markdown으로 작성
- 제목, 소제목, 본문, 코드블록 구조를 자연스럽게 사용할 것
- 기본은 문단형 설명으로 가고, 꼭 필요한 경우에만 bullet을 사용할 것
- bullet만 잔뜩 나열하는 문서 스타일은 피할 것

## 14. 최종 작성 원칙 요약

앞으로 내가 블로그용 글 정리를 요청하면 아래 원칙을 우선 적용한다.

- 내 블로그 스타일처럼 설명형 문체로 쓸 것
- 개념 -> 구조 -> 구현 흐름 -> 핵심 코드 -> 결과 순서로 정리할 것
- 코드 전체를 다 넣지 말고 핵심 부분만 발췌할 것
- 코드보다 설명 비중을 더 높일 것
- 검색 키워드를 제목, 첫 문단, 소제목, 본문에 자연스럽게 넣을 것
- "무엇을 했다"보다 "왜 이렇게 설계했는지"를 더 강조할 것
- 마지막에는 현재 결과와 다음 단계까지 연결해서 마무리할 것

## 15. 짧은 명령 예시

이 문서를 기준으로 나중에 아래처럼 요청하면 된다.

### 예시 1

`지금까지 작업한 내용을 blogSEO.md 기준으로 블로그 글처럼 정리해줘`

### 예시 2

`이번 Player 이동/점프 구현 내용을 내 블로그 말투와 SEO를 섞어서 포스팅 초안으로 써줘`

### 예시 3

`핵심 코드만 발췌해서, 왜 이렇게 구조를 나눴는지 중심으로 블로그 글 써줘`
